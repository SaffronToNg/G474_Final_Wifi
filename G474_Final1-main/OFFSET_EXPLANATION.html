偏置在本项目中的意义与实现说明

适用对象：当前 G474 迁移工程与原 F334 电源控制工程。

目的：解释“为什么这个项目必须做偏置（offset）处理”、“偏置解决什么问题”、“原工程怎么做的”、“当前 G474 版本是如何最小恢复的”。

1. 什么是偏置

在这个项目里，偏置（offset）主要针对的是电流相关采样通道：

  • IL：电感电流相关通道
  • IY：Y 侧电流相关通道
  • IX：X 侧电流相关通道

这些通道并不是“零电流时 ADC 直接读 0”，而通常是通过模拟前端加了一个中点电压，使得：

  • 零电流时，ADC 原始值大约落在 12 位 ADC 中间附近，例如 ~2048
  • 有正向/反向电流变化时，ADC 值围绕这个中点上下偏移

所以 raw 值本身不等于“真实电流”，raw 值只是“中点偏置 + 真实信号”的组合。

2. 为什么这个项目必须做偏置

如果不做偏置处理，后面所有依赖电流判断的逻辑都会有问题。

2.1 不做偏置会发生什么

  • 零电流时，程序会误以为一直有一个很大的“假电流”
  • 小电流判断会漂移
  • 过流保护门限会失真
  • 控制环会带着直流偏差工作
  • 模式切换（例如 BUCK / BOOST）时电流方向判断会不可靠

2.2 为什么不能直接假设 offset 固定等于 2048

  • 运放有失调
  • 电阻分压/采样网络有误差
  • ADC 本身有误差
  • PCB 与器件温漂会造成零点变化

所以原工程的正确思路不是“写死一个中点值”，而是：上电后在静态条件下，实际测出每一路自己的零点。

3. 偏置在原工程里是怎么做的

原工程在初始化状态阶段，会做一个非常关键的动作：

  1. 连续累计 256 次电流通道 raw 值
  2. 分别求出 IxOffset、IyOffset、ILOffset
  3. 后续采样时，用 raw - offset 得到去偏置值

核心思想可以概括为：

offset = 256 次静态采样的平均值
zero_based = raw - offset

这样做的结果是：

  • 静态无电流时，zero_based ≈ 0
  • 真正有电流变化时，zero_based 才偏离 0

4. 当前 G474 迁移版本是怎么实现的

当前 G474 版本里，我没有把原工程完整状态机直接搬过来，而是先恢复了“最小偏置框架”。

4.1 当前恢复了哪些字段

字段 含义
OFFN当前已经累计了多少次偏置样本
OFFRDY偏置是否已经计算完成，0=未完成，1=完成
ILOIL 通道 offset
IYOIY 通道 offset
IXOIX 通道 offset
ILZIL 去偏置结果（zero-based）
IYZIY 去偏置结果（zero-based）
IXZIX 去偏置结果（zero-based）

4.2 当前实现逻辑

  1. 每次主循环都执行语义更新与处理更新
  2. 如果偏置样本数还没到 256，就继续累计 IL / IY / IX raw 值
  3. 当样本数到 256 时，求平均值得到 offset
  4. offset 就绪后，持续计算 zero_based = raw - offset

5. 为什么现在看到的结果说明“偏置层通了”

你目前已经看到类似结果：

OFFN=256 OFFRDY=1
ILO=2041 IYO=2027 IXO=2023
ILZ=-3~+2 IYZ=-1~+4 IXZ=-2~+4

这组结果说明：

  • OFFN=256：偏置累积完整完成
  • OFFRDY=1：偏置已经建立成功
  • ILO/IYO/IXO 在 2000 附近：符合中点偏置通道特征
  • ILZ/IYZ/IXZ 在 0 附近小幅波动：说明去偏置成功，零基准已建立

因此可以判定：当前偏置初始化框架已打通。

6. 为什么偏置阶段不能乱接 3.3V

在偏置学习阶段，不建议对 IL / IY / IX 这三路电流通道强行接 3.3V。

因为偏置学习的目标不是“验证某一路输入会不会变”，而是：

在无真实电流激励时，学到各电流通道自己的零点

如果你在这个阶段人为给这些通道施加强输入，学出来的 offset 就会偏掉，后面去偏置结果就不再可信。

正确做法：让系统保持静态、无真实电流激励，再完成 offset 学习。

7. 偏置在原工程后续要用来做什么

偏置不是为了“看着好看”，而是后面很多逻辑的前提：

  • 电流方向判断
  • BUCK / BOOST 模式下的正负号转换
  • 过流保护
  • 短路检测
  • 控制环电流反馈

如果 offset 不准，那么这些逻辑全都会被带偏。

8. 当前实现和原工程完整实现的差别

当前 G474 版本只恢复了“最小偏置框架”，还没有恢复完整行为。

项目 当前 G474 版本 原工程完整版本
偏置累积已恢复已具备
zero-based 去偏置已恢复已具备
BUCK/BOOST 方向翻转未恢复有
控制意义上的电流值未恢复完整有
状态机联动未恢复有
保护/控制环使用未恢复有

也就是说，现在只是把“偏置底座”搭起来了，还没把“偏置参与业务逻辑”全部恢复。

9. 当前阶段的通过标准

如果满足以下条件，就可以认为偏置阶段通过：

  • OFFN=256
  • OFFRDY=1
  • ILO / IYO / IXO 稳定
  • ILZ / IYZ / IXZ 接近 0
  • 串口、TIM、ADC、HRTIM 其它观测项仍稳定

目前你的工程已经满足这些条件，因此可认为“偏置/初始化采样框架”已经打通。

10. 下一步会做什么

偏置层打通后，后面更合理的方向通常是：

  1. 恢复目标输入 / 标定层（例如 VADJ / IADJ 目标处理）
  2. 恢复模式相关采样处理
  3. 恢复状态机骨架
  4. 最后才接控制环和输出行为

不建议现在直接跳去恢复 PWM 输出或闭环控制。
