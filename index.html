系统组成概览

从当前代码可以抽象出如下系统结构：

  • 采样链：ADC1 / ADC2 + DMA，结合功率板采样电路获取电压、电流、调节量
  • 调度链：TIM2 / TIM3 周期任务驱动采样处理、状态机、控制与串口输出
  • 驱动链：HRTIM 负责功率相关输出驱动
  • 状态机：Buck / Boost 下共用 WAIT / RISE / RUN 概念，但实现细节不同
  • 交互链：KEY2 触发输出使能，USART3 发送状态帧并接收 V#### 设置 VTAR，TFT 显示最终数值
  • 显示 / 校准链：TFT 与串口上报值是经过换算、补偿、偏置修正后的结果

状态机流程（概念）

WAIT

等待 offset / 模式识别稳定，并等待按键或条件触发。此时系统不应被简单理解为“已经输出”。

RISE

逐步建立驱动与占空比，Buck / Boost 的过渡过程都在这里发生。KEY2 参与触发输出使能。

RUN

进入闭环或稳定运行阶段。此时 TFT 与串口上报的是最终显示链值，而不是裸 ADC 原始值。

串口协议速览

代表性工程的代码显示，USART3 会周期发送状态帧，并接收固定格式命令：

0xAA
mode
vin
vout
ii
io
vtar
XOR

设置命令使用：

V####，例如 V2500

其作用是把目标显示值反推为系统内部的 vref_target。

高风险误区

  • 不要默认三个子工程完全等价。 它们共享骨架，但参数、校准、行为细节可能不同。
  • 不要默认该仓库可直接跑在任意 STM32G474 开发板上。 当前工程明显依赖板级功率电路、TFT、KEY2、HSE、采样链。
  • 不要把 TFT / 串口值直接当作瞬时物理量。 这些数值可能已经经过滤波、偏置修正、经验补偿和显示换算。
