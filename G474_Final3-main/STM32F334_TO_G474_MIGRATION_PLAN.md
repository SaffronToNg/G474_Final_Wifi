# STM32F334 → STM32G474 分阶段迁移计划

> 适用范围：仅允许修改 `G474yizhi1` 文件夹内文件。
>
> 约束：**禁止修改原 F3 工程**，原工程仅作为对照参考。
>
> 目标：在 `G474yizhi1` 新框架上，分阶段、可验证地恢复原工程功能，避免一次性整体迁移导致无法定位问题。

---

## 1. 迁移原则

### 1.1 不可违反的约束

- [ ] 原工程目录只读参考，不做任何修改
- [ ] 所有新增/修改文件仅位于 `G474yizhi1` 内
- [ ] 每次只迁移一个功能块
- [ ] 每迁移一部分，必须先验证，再继续下一部分
- [ ] 未验证通过前，不进入下一阶段

### 1.2 迁移总体策略

本次迁移不以“复制文件数量”为进度标准，而以“关键链路是否闭合并验证”为标准。

优先按以下链路推进：

1. 采样链：ADC -> DMA -> 缓冲区 -> 数据解析
2. 调度链：TIM2 / TIM3 / HRTIM 中断 -> 对应业务函数
3. 驱动链：HRTIM -> PWM 输出 -> Compare 更新
4. 控制链：采样 -> 控制器 -> 占空比
5. 保护链：OCP / OVP / UVP / OTP -> 关断 / 状态机响应
6. 交互链：USART / 按键 / LCD

---

## 2. 当前工程认知

## 2.1 旧工程（F3）角色定位

旧工程不是简单的 HAL 外设工程，而是：

- CubeMX/HAL 外设层
- 自定义控制层
- 中断驱动状态机
- HRTIM 高频控制环
- 串口协议与 LCD 显示

旧工程的关键应用层文件包括：

- `function.c/.h`
- `CtlLoop.c/.h`
- `RS232.c/.h`
- `LCDshow.c/.h`
- `LCDDriver.c/.h`

## 2.2 新工程（G474）角色定位

当前 `G474yizhi1` 仍是硬件框架工程，已具备：

- ADC1 / ADC2
- HRTIM1
- TIM2 / TIM3
- USART3
- DAC1
- DMA
- GPIO

但目前还未具备：

- 旧工程应用层
- 旧工程中断执行语义
- 原控制状态机
- 原串口协议处理
- 原控制环
- 原内存区放置策略

**结论：当前新工程可作为迁移底座，但不是可直接运行原功能的等价工程。**

## 阶段 1：启动序列迁移（第一优先级）

### 目标
让 G474 工程具备与旧工程接近的启动顺序，但**先不进入完整闭环控制**。

### 拟修改范围（仅 G474 工程）

- `Core/Src/main.c`
- 必要时新增占位头文件/源文件到 `Core/Inc`、`Core/Src`

### 计划迁移内容

- [ ] 增加 ADC1 校准
- [ ] 增加 ADC2 校准
- [ ] 启动 ADC1 DMA
- [ ] 启动 ADC2 DMA
- [ ] 启动 DAC
- [ ] 启动 USART3 接收机制
- [ ] 启动 TIM2 中断
- [ ] 启动 TIM3 中断
- [ ] 启动 HRTIM 计数
- [ ] 明确 PWM 输出初始策略（默认关闭或受控开启）
- [ ] 删除/替换当前 `while(1)` 中无关测试逻辑

### 本阶段明确不做

- [ ] 不迁移 `LoopCtl()`
- [ ] 不恢复完整状态机
- [ ] 不恢复完整保护逻辑
- [ ] 不接入 LCD 逻辑
- [ ] 不直接进入闭环

### 验证标准

- [ ] ADC DMA 缓冲区持续更新
- [ ] TIM2 中断正常进入
- [ ] TIM3 中断正常进入
- [ ] HRTIM 中断可进入或可观测
- [ ] USART3 接收链路可被触发
- [ ] 系统运行稳定，无异常 HardFault

### 风险点

- HRTIM / ADC / DMA 启动顺序不同可能导致空数据或异常中断
- USART 接收机制不能简单照抄旧工程，需结合 G4 当前 HAL 结构确认

### 阶段 1 详细执行方案

> 原则：本阶段只建立“启动链路”，不建立“控制链路”。
>
> 只允许修改 `G474yizhi1` 内文件；旧 F3 工程仅用于对照，不做任何改动。

#### 1. 计划修改文件

- `Core/Src/main.c`
- `Core/Inc/main.h`（仅当需要增加调试宏或占位声明时）
- `Core/Src/stm32g4xx_it.c`（仅增加最小化观测代码，不接入旧业务逻辑）
- `Core/Src/usart.c`（仅在需要补充接收启动配套逻辑时修改）
- `Core/Src/hrtim.c`（仅在需要明确初始输出策略时修改）

#### 2. 执行顺序（必须按顺序推进）

##### 步骤 1：清理当前临时测试主循环

**目标**：移除当前“每秒发 A + 翻转 IO”的临时验证逻辑，避免其掩盖后续真实启动行为。

**具体动作**：

- [ ] 删除或注释 `while(1)` 中串口发送字符 `A` 的测试逻辑
- [ ] 删除或注释 `PC14` 翻转测试逻辑
- [ ] 将主循环改为最小空转结构，保留后续插入状态观测点的空间

**改动范围建议**：

- `Core/Src/main.c`

**完成判定**：

- [ ] 主循环不再包含与迁移目标无关的测试输出
- [ ] 工程仍能正常编译下载

##### 步骤 2：按旧工程顺序补齐启动动作

**目标**：让 G474 启动过程尽量接近旧工程，但暂不进入应用层逻辑。

**建议顺序**：

1. `HAL_Init()`
2. `SystemClock_Config()`
3. 外设初始化（保留当前 CubeMX 初始化顺序，若需要再调整）
4. ADC1 校准
5. ADC2 校准
6. 启动 ADC1 DMA
7. 启动 ADC2 DMA
8. 启动 DAC
9. 启动 USART3 接收
10. 启动 HRTIM 计数
11. 启动 TIM2 中断
12. 启动 TIM3 中断

**注意**：

- [ ] 本阶段不要求完全复制旧工程初始化顺序，但必须记录最终采用顺序
- [ ] 若某步骤依赖占位缓冲区或最小全局变量，可以先在 G474 工程内新增最小占位定义
- [ ] 所有占位对象必须明确命名，禁止使用含糊的临时魔法变量

##### 步骤 3：建立最小 DMA 观测能力

**目标**：不迁移 `function.c` 的前提下，先确认 ADC DMA 链已经真实工作。

**具体动作**：

- [ ] 在 G474 工程内定义最小 ADC1/ADC2 DMA 缓冲区
- [ ] 缓冲区长度保持与旧工程一致：ADC1 为 5，ADC2 为 3
- [ ] `HAL_ADC_Start_DMA()` 的目标地址明确指向这两个缓冲区
- [ ] 主循环或调试输出中仅做“是否更新”的轻量观测，不加入数据处理逻辑

**建议**：

- [ ] 本阶段先不引入平均值/偏置/物理量换算
- [ ] 本阶段先只验证“DMA 数据在变”与“长度顺序正确”

##### 步骤 4：建立最小 USART 接收链

**目标**：确认串口接收链条存在，但不急于恢复原协议状态机。

**具体动作**：

- [ ] 在 G474 工程内定义最小接收缓冲区
- [ ] 选择一种清晰策略：
  - `HAL_UART_Receive_IT()` 单次接收并在回调/中断后重装载
  - 或等后续协议迁移时再升级为完整状态机
- [ ] 本阶段只验证“收到字节后系统能感知”，不验证完整协议

**禁止事项**：

- [ ] 不要现在就把旧 `RS232.c/.h` 整包搬入
- [ ] 不要现在就把 USART 中断改成复杂业务分发

##### 步骤 5：建立最小中断可观测性

**目标**：确认 TIM2 / TIM3 / HRTIM 中断真的进入。

**具体动作**：

- [ ] 为 TIM2 / TIM3 / HRTIM 各准备一个最小计数器或标志位
- [ ] 在对应 IRQ 或 HAL 回调中只做“计数 +1 / 标志置位”这类轻量动作
- [ ] 通过串口打印、断点观察或监视窗口确认计数在变化

**注意**：

- [ ] 本阶段不要把旧工程 ISR 里的业务逻辑塞进来
- [ ] 本阶段不要为了“像旧工程”而过早改为复杂裸寄存器清中断方案

##### 步骤 6：明确 HRTIM 初始输出策略

**目标**：防止启动阶段误输出危险 PWM。

**具体动作**：

- [ ] 确认本阶段仅启动 HRTIM 计数，不默认启动功率输出
- [ ] 若需要显式关闭 TA1/TA2，则在代码中明确写出而不是依赖默认状态
- [ ] 在文档中记录当前采用的是“只计数、不出功率波形”还是“输出已关断等待后续状态机接管”

**判定标准**：

- [ ] 上电后不会因为初始化顺序导致意外 PWM 导通

#### 3. 阶段 1 验证方法

### 目标

## 4. 文件迁移策略

## 4.1 直接参考但不应立即整体复制的文件

以下文件虽重要，但不建议第一阶段整体复制：

- `function.c/.h`
- `CtlLoop.c/.h`
- `RS232.c/.h`
- `stm32f3xx_it.c`

原因：

- 与中断模型深度耦合
- 与旧芯片寄存器、内存布局、启动顺序耦合
- 直接复制会增加定位难度

## 4.2 可分块提取的逻辑

优先按函数级迁移：

- 采样处理函数
- 状态机函数
- 协议函数
- 控制函数
- 显示函数

不要按“整文件整包迁移”作为默认策略。

## 5. 风险清单

## 5.1 高风险项

- [ ] HRTIM 中断执行模型不一致
- [ ] ADC 触发相位改变
- [ ] DMA 数据顺序错位
- [ ] 闭环参数直接复用导致控制异常
- [ ] 内存段 `CCMRAM` 在 G474 上未适配
- [ ] GPIO / 板级功能资源未完全映射

## 5.2 中风险项

- [ ] 串口协议接收模型差异
- [ ] TIM 中断节拍与旧工程不完全一致
- [ ] 保护阈值与新采样比例不完全一致

## 5.3 低风险项
