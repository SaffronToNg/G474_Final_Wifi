TFT 采样-语义-显示判定文档

目的：回答“当前屏幕显示的电流值，是否已经是 ADC 采样后映射得到的实际物理电流值；各显示项分别对应什么”。

判定范围：仅基于当前工程中已检查到的显示链路、采样语义链路、最小处理链路和目标/保护链路做证据化判断；不臆造未看到的硬件系数或外部标定文档。

1. 关键结论

结论 1：当前 TFT 上的 IOUT 不能直接判定为“已经严格映射到真实安培值的实际输出电流”。

当前代码能够确认的是：

  1. 先从 ADC DMA 缓冲区取到某一路电流原始码值；
  2. 再做 offset 去零；
  3. 再做简单平均；
  4. 最后乘一个固定经验比例 150/227 用于 TFT 显示。

但在当前检查到的这条 TFT 显示链路中，没有看到“电流采样电阻 + 放大倍数 + ADC 参考 + 通道标定系数”这一套显式的安培换算闭环。因此更稳妥的说法是：它是输出侧电流通道的去零后显示量，而不是已经被严格证明的真实物理电流值。

结论 2：TFT 显示的 IOUT 是“模式语义量”，不是固定物理通道名。

  • BUCK 模式：IOUT = Y 侧电流通道（Iy）
  • BOOST 模式：IOUT = X 侧电流通道（Ix）

结论 3：当前 TFT 上的 VIN 口径并不完全统一。BUCK/BOOST 下使用的变量口径不同，不能无条件把它都当成“统一定义的实际输入电压值”。

2. ADC 原始语义映射

当前采样语义映射在 Core/Src/sample_semantics.c 中已经明确写出：

ADC1_RESULT[0] -> IL
ADC1_RESULT[1] -> Vy
ADC1_RESULT[2] -> Iy
ADC1_RESULT[3] -> Vx
ADC1_RESULT[4] -> Ix

ADC2_RESULT[0] -> Vadj
ADC2_RESULT[1] -> Iadj
ADC2_RESULT[2] -> Temp
DMA / ADC 槽位 变量名 当前含义
ADC1[0]IL电感电流相关通道原始码值
ADC1[1]VyY 侧电压原始码值
ADC1[2]IyY 侧电流原始码值
ADC1[3]VxX 侧电压原始码值
ADC1[4]IxX 侧电流原始码值
ADC2[0]Vadj电压调节输入原始码值
ADC2[1]Iadj电流调节输入原始码值
ADC2[2]Temp温度相关原始码值

这一层是“物理/采样通道语义”，还不是输入/输出语义。

3. 当前样本处理链路

在 Core/Src/sample_processing.c 中，当前最小恢复链路主要做了三件事：

  1. raw 直取：把语义层的 raw 码值放进 g_sample_processed
  2. offset 学习：前 256 次累计电流通道，得到 ix_offset / iy_offset / il_offset
  3. 去零与平均：得到 ix_zero_based / iy_zero_based / il_zero_based 以及 vx_avg / vy_avg / ix_avg / iy_avg
zero_based = raw - offset
avg = 一阶平滑后的平均 ADC 值

要点：这里恢复的是“去偏置 + 平均”框架，不等于“已经完成工程级物理量校准”。代码注释也明确说当前阶段不恢复完整电流方向语义和完整校准行为。

4. TFT 三个主显示项分别对应什么

屏幕项 BUCK 模式 BOOST 模式 判定
VTAR (vref_target * 6800) >> 12 (vref_target * 6800) >> 12 目标值，不是实时采样值
VIN (vx_avg * 6800) >> 12 (vy_calibrated * 6800) >> 12 模式语义下的输入电压显示，但 BUCK/BOOST 使用变量口径不一致
IOUT ((iy_avg - iy_offset) * 150 + 113) / 227 ((ix_avg - ix_offset) * 150 + 113) / 227 模式语义下的输出侧电流显示量，不足以直接证明是真实安培值

5. 为什么当前 IOUT 不能直接叫“实际电流值”

5.1 已经看到的显示公式

BUCK:  iout_feedback = iy_avg - iy_offset
BOOST: iout_feedback = ix_avg - ix_offset

if (iout_feedback < 0) iout_feedback = 0;

iout_display = (iout_feedback * 150 + 113) / 227

5.2 这个公式说明了什么

  • 已经做了去偏置，说明它不是纯 raw；
  • 已经做了模式映射，说明它在表达“输出侧”而不是固定 X/Y 侧；
  • 但最后只看到一个固定比例 150/227，没有看到显式电流物理换算闭环。

5.3 因而更稳妥的命名

当前更稳妥的描述应是：TFT 的 IOUT 是输出侧电流通道经 offset 去零、平均后，再经过经验比例压缩得到的显示值。

不稳妥的描述则是：“屏幕此时显示的就是已经严格校准的真实输出电流安培值。”

6. VIN 为什么也不能掉以轻心

当前 TFT 中，VIN 的模式实现并不对称：

  • BUCK：直接使用 vx_avg 参与显示
  • BOOST：使用 vy_calibrated 参与显示

但在 Core/Src/target_calibration.c 中，vx_calibrated / vy_calibrated 本来都各自有校准路径。

这意味着当前 TFT 的 VIN 并不是“统一通过同一口径校准后的输入电压显示”，而是存在 BUCK/BOOST 两套口径混用的情况。

7. TFT 数字格式是什么

在 Core/Src/LCDDriver.c 中，LCDShowFnum() 的显示格式固定为：

digits[0] digits[1] . digits[2] digits[3]

例如传入 1234，会显示成：

12.34

因此：

  • VTAR、VIN、IOUT 都是按 两位整数 + 两位小数 的格式显示；
  • 显示函数本身并不知道单位物理意义，只负责插入小数点；
  • 真正决定量纲的是上层传入的换算公式。

8. 串口与 TFT 的语义口径并不相同

在 Core/Src/main.c 的状态打印里，注释明确写着：

VX/VY 始终是 X/Y 侧电压
IX/IY 始终是 X/Y 侧电流
上位机根据模式 M 自行判断哪侧是输入、哪侧是输出

这说明：

  • 串口状态输出：偏向物理语义（X/Y）
  • TFT 显示：偏向模式语义（VIN / IOUT）

所以不能直接拿串口的 IX/IY 与 TFT 上的 IOUT 逐字面对照，而不先做模式映射。

9. 最终判定表

问题 判定 理由
TFT 上的 IOUT 是否等于已经严格映射的实际电流值？ 不能直接确认 当前只看到去偏置、平均和经验比例显示，未在这条显示链路中看到完整的安培标定闭环
IOUT 是否有明确物理来源？ 可以确认 BUCK 取 Iy，BOOST 取 Ix，都是输出侧电流语义映射结果
VTAR 是否是实时采样值？ 不是 它来自目标值 vref_target 的比例换算
VIN 是否口径统一？ 当前不统一 BUCK 用 vx_avg，BOOST 用 vy_calibrated

10. 本文档最重要的一句话

当前 TFT 的 IOUT 更像“输出侧电流通道的去零经验显示值”，而不是已经被代码充分证明的“真实输出电流安培值”。

但它的模式对应关系是明确的：BUCK 看 Iy，BOOST 看 Ix。
